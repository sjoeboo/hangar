Decision 4 of 4

Mouse Support Scope

Interaction

Bubble Tea (the TUI framework hangar uses) supports mouse events — you just have to opt in and handle them. Mouse support already exists for the PR overview table. The question is how far to extend it for the new nav tabs and filter pills. More clickable zones = more implementation work + more edge cases to test. Where's the right stopping point?

Only the three nav tabs (Sessions, PRs, Todos) respond to mouse clicks. Everything else — filter pills, session list items, preview pane — stays keyboard-only. This is the minimum viable mouse addition: just make the new navigation elements clickable.

Works well when

  • Smallest implementation surface, fewest edge cases
  • Keyboard users aren't impacted at all
  • You can always expand mouse support later

Watch out for

  • Filter pills look clickable (labeled pills) but aren't
  • Inconsistent — some rows are clickable, others aren't

Both the nav tabs and the status filter pills are mouse-clickable. This is the natural choice: since we're relabeling the filter pills to look like interactive controls (Option A from Decision 3), they should actually behave as interactive controls. Click Waiting to filter to waiting sessions; click Sessions to switch views. Session list items stay keyboard-driven.

Works well when

  • UI affordance matches behavior (labeled pills = clickable)
  • Mouse users can do the most common navigation actions
  • Consistent: all controls in the top two rows are clickable

Watch out for

  • Requires hit-testing both rows of elements
  • Need to guard against accidental click during scroll

Tabs, filter pills, and session list items are all mouse-responsive. Single-click moves the cursor to that session (same as pressing ↑/↓). Double-click attaches (same as pressing Enter). This is how tools like lazygit and k9s handle their list views. Makes hangar feel much more like a normal GUI app.

Works well when

  • Users expect mouse to work everywhere
  • Sessions list is long and scrolling by mouse is natural
  • Consistent with how k9s, lazygit behave

Watch out for

  • Hit-testing session rows requires tracking render positions
  • Mouse scroll vs click disambiguation in Bubble Tea needs care
  • More implementation work — medium complexity

Option D
Chosen

Full Mouse Mode

1 Click tab → switch view
↓
2 Click filter pill → filter list
↓
3 Click session → select it
↓
4 Scroll wheel → scroll list
↓
5 Double-click → attach session
Everything is mouse-aware: tabs, filter pills, session list (click to select, double-click to attach), and scroll wheel to scroll the session list without needing to click first. This is the most complete mouse experience — you could theoretically never touch the keyboard. But it's also the most implementation work.

Works well when

  • You want hangar to feel like a fully GUI-capable app
  • Users frequently switch between mouse and keyboard
  • Scroll wheel support makes long session lists much easier

Watch out for

  • Scroll wheel in Bubble Tea conflicts with terminal scrollback
  • Highest implementation complexity and test surface
  • tmux mouse mode must also be enabled for scroll to work
