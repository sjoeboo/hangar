Decision 1 of 4

Tab Navigation Model

Information Architecture

Right now, the PR overview is hidden behind the P key — you have to already know it exists. A visible tab bar at the top would make it discoverable. This decision is about what those tabs are and how they work: do we just add Sessions + PRs, or do we design something more extensible for the future (todos, logs, settings)? The answer shapes everything else.

You add exactly two tabs: Sessions and PRs. That's it. Switch between them with Tab/Shift+Tab or by clicking. The P key can still work as a shortcut. This is the smallest change that solves the discoverability problem without any over-engineering.

Works well when

  • You want the smallest possible change
  • Sessions + PRs cover 95% of use cases
  • Simpler to implement and maintain

Watch out for

  • Adding a 3rd view later requires rethinking this
  • No obvious place for future views (todos full-screen, logs, etc.)

Build a proper tab bar with Sessions, PRs, and Todos from day one. The todos dialog currently opens as an overlay — promoting it to a full tab makes it a first-class view. The tab system is designed to accept new entries easily. Navigate with Tab/Shift+Tab or number keys (1/2/3).

Works well when

  • You want to keep adding features as proper views
  • Todos as a tab is more discoverable than t key
  • Future views slot in without rework

Watch out for

  • Slightly more implementation work upfront
  • Tab bar gets crowded if you add 5+ views

Instead of full tabs, show a breadcrumb at the top — Hangar › Sessions — with small hints for available views like [P] PRs. Keeps the existing keyboard shortcuts but makes them visible. Less visual overhead, but the views still feel somewhat hidden.

Works well when

  • You want minimal visual footprint
  • Keyboard-first users who already know shortcuts
  • Easiest to implement

Watch out for

  • Still not very discoverable for new users
  • Doesn't work as well for mouse navigation
  • Doesn't solve the "I didn't know PRs existed" problem

⎇ PRs
☑ Todos
SESSIONS
● feature-auth
◐ fix-login-bug
○ data-pipeline
A narrow left sidebar holds the view navigation (Sessions, PRs, Todos) with icons. The main content area stays the same. This is how tools like lazygit organize their multiple views. Gives each view an icon + label and is very clickable with the mouse.

Works well when

  • You plan to add many more views (5+)
  • Icon + label is clearer than tab text alone
  • Works great with mouse navigation

Watch out for

  • Costs ~10 columns of horizontal terminal space
  • Overkill for just 2-3 views
  • Significantly more layout work to implement
